Secure Configuration and Version Control (7.2)

Controlled configurations and versioning assist in the secure deployment, execution, and maintenance of software systems using a structured change management process. This process is sometimes generally referred to as Security Configuration Management (SecCM) or the management and control of configurations for an information system to enable security and facilitate risk management. A software system is typically composed of multiple components, which are updated and reconfigured at differing rates in response to business needs for enhancements or sustainment.

Learning Objectives

  • Describe Security Configuration Management (SecCM).

Overview

NIST 800-128—Guide for Security-Focused Configuration Management of Information Systems describes the need for configuration management to protect software and systems. The need for changes may arise based on hardware updates, defect amelioration, new security hardening needs, or changing business functions. Software updates that require configuration changes on the host system or within the software configuration are common. The combination of system changes at multiple levels (e.g., OS configuration, application, and application configuration) increases complexity and the likelihood of unintentionally introducing production issues. To safeguard against such an occurrence, configuration management should be a well-defined, controlled process.

Configuration Management

Configuration Management (CM) encompasses establishing and maintaining the integrity of software products and systems by controlling processes for adding, modifying, and monitoring configurations. Each item under change control is essentially a configuration item. A Configuration Item (CI) such as software, firmware, hardware, or documentation is a discrete unit of the organization’s configuration control processes. A CM Plan describes roles, policies, and procedures required to effectively execute CM for CI updates. CM Plans typically include:

  • Change Control Board (CCB). This is composed of parties responsible for controlling and approving changes throughout the life cycle software systems. 
  • Configuration Item Identification. This covers selection and the naming method for CI.
  • Configuration Change Control. This is the process for managing baseline configuration changes.
  • Configuration Monitoring. This is the process for assessing baseline compliance that includes reporting.

Baseline Configuration

A baseline configuration is a set of reviewed and accepted configuration items for a system that becomes the basis for builds, releases, and production conformance. Baselines are formally reviewed and accepted before being adopted. Any new changes to the baseline must undergo change control procedures. Baselines and corresponding CIs can change based on time and the system-development life cycle (SDLC) phase.

Early in the SDLC system, baselines may be functional requirements. As the system matures, the baseline will likely include software design artifacts and component system configurations. Baselines may be organizationally established for various computing environments that are distinguished by classification level (e.g., Secret vs. Unclassified) or life cycle phase (e.g., test vs. production). When a new baseline for a computing environment or system is established, the old baseline is deprecated but archived. The CCB may reverse an approval decision if quality or security issues are discovered. If this occurs, a previous baseline would be used to rollback as needed. NIST SP 800-53 security control CM-2 Baseline Configuration describes the SecCM phase for identifying and implementing secure configurations.

Information Systems and Hardware Configuration

Secure configurations are implemented to reduce operational system risk. Configuration may enforce trusted software installation, patch maintenance, or security-minded configuration values. Secure configurations for software systems commonly reference integrated supplier software products. The concept of secure configuration is also relevant to hardware. An organization must track and label assets. Within the context of a traditional on-premises server room, a rack of servers might have a particular configuration. The baseline configuration of the server rack might enforce equipment redundancy and support effective cable management and other organizationally defined requirements. NIST SP 800-53 security controls CM-6 Configuration Settings and CM-7 Least Functionality also support identifying and implementing secure configurations within SecCM.

Configuration Change Control

Configuration change control is the formally documented process for controlling additions, modifications, and removal of system configurations or associated CIs. Configuration change control for software and systems typically includes:

  1. Proposal
  2. Justification
  3. Implementation
  4. Test/Evaluation/Review
  5. Change Disposition

Configuration change control should apply to all change scenarios—whether an emergency or unscheduled business need or routine updates. Changes should be managed from proposal through all steps including change disposition. Each step should be clearly defined and documented. NIST SP 800-53's security control CM-3 Configuration Change Control and CM-5 Access Restrictions for Change both support the concept of controlling configuration changes within SecCM.

Configuration Documentation

SecCM policies should establish expectations for the organization using specific, measurable, and testable objectives. In general, configuration management policies explicitly define configuration management organizational goals and requirements. SecCM policy documents describe organizational objectives informing the creation of the steps required to execute change management policy goals. SecCM procedures specifically focus on the methodology to propose, justify, implement, and test changes utilizing the concepts of a CCB, baseline, and CI.

The CM Plan is operational and describes how SecCM policy (i.e., strategy) should be implemented. The SecCM Plan might be incorporated into the CM Plan or be kept separate. The SecCM Plan may be enterprise-wide or apply to a specific group (e.g., software development, DevSecOps, infrastructure) within the organization. Bottom line, organizations should tailor configuration document implementation to integrate with and support business processes. The SecCM Plan and other documents are initiated during SecCM planning and support the implementation of both NIST SP 800-53's security controls CM-1 Configuration Management Policy and Procedures and CM-9 Configuration Management Plan.
