Domain 2 Key Takeaways

If you remember nothing else...

Risk management is a continuous and iterative process that is best executed when secure software life cycle management has the same characteristics—continuous and iterative.

Security practitioners must understand the difference between operational security for software infrastructure, secure software development, and securing the development pipeline.

Summary

Overall, security practitioners reviewing CSSLP Domain 2 need to understand the importance of various software development methodologies (e.g., Waterfall, Agile, Spiral, etc.) that have been introduced over the years to meet the evolving needs of businesses. Agile methodologies help with the velocity of development, but security must be adapted to short development iterations. While DevOps and CI/CD are valuable in generating organizational returns, DevSecOps attempts to bridge the gap between DevOps and security. Agile development fosters continuous improvement, which creates opportunities for teams to regularly and frequently reflect on performance.

Compliance adds further complications, as most organizations are subject to a combination of industry-specific, state, national, or international regulations, laws, standards, or contractual obligations. Compliance can be seen as an organizational risk. Risk management is a continuous and iterative process that is best executed when secure software life cycle management has the same characteristics—continuous and iterative.

Continuous improvements in software-development life cycle (SDLC) processes are recommended, regardless of which software development process is adopted by an organization. Many factors determine an organization’s secure software life cycle management. At an enterprise level, organizational risk tolerance will play a role in determining funding and focus for security within software development teams. On a more micro level, software development team culture and individual contributor experiences with security are examples that may shape the implementation of security with respect to software development.

The most important takeaway is that security practitioners must understand the difference between operational security for software infrastructure, secure software development, and securing the development pipeline. Each of these areas of software security has life cycle considerations that security practitioners must observe, integrate safeguards, and establish monitoring and reporting. Failure to understand and implement security solutions in each phase of the life cycle will result in a lack of organizational cyber resiliency and the realization of risks.

Managing security within a software development methodology

  • A predictive methodology relies on known objectives, paths, and flows from one phase of development (software maturation) to the next.
    • Phases are sequential and planning is based on known expected outcomes such as specific project milestones.
    • A classic example of predictive methodology would be Waterfall.
  • An adaptive/iterative methodology relies on short iterations where the project is deconstructed into small components that may be started and completed during short development iterations.
    • A classic example of adaptive methods includes the combination of Agile and Scrum concepts.
    • Development Operations (DevOps) is often described as a logical extension of Agile, pioneered by lean-thinking organizations seeking solutions for rapid and frequent delivery of software.
    • Development, Security, and Operations (DevSecOps) is everything that DevOps is but adds an additional focus on software assurance.
    • DevSecOps bridges the gap between security and DevOps. 

Identifying and adopt security standards 

  • Center for Internet Security (CIS) benchmarks are being adopted by public and private entities globally as the best-practice security configuration guides.
  • Two different configuration profiles for CIS benchmarks:
    • Level 1: The intent of the Level 1 profile benchmark is to lower the attack surface of your organization while keeping machines usable and not hindering business functionality.
    • Level 2: The recommendations associated with the Level 2 profile can adversely affect an organization if not implemented appropriately or without due care.
  • Configuration standards in the United States for DoD systems require compliance with a technical testing and hardening framework known as Security Technical Implementation Guide (STIG).
  • STIG compliance category levels:
    • Category I includes any vulnerability that will result in the loss of CIA and may result in loss of life, facility, or mission. It is a mission failure.
    • Category II includes any vulnerability that may result in the loss of CIA. It may also result in personal injury or equipment damage. This is mission degradation. 
    • Category III includes any vulnerability that degrades security measures protecting CIA.
  • Security Content Automation Protocol (SCAP) is a way of automating STIG checking.
  • Common Criteria (CC) allows for the comparability of results between independent security evaluations.
  • Common Weakness Enumeration (CWE) that describes a taxonomy for identifying the common sources of software flaws (e.g., buffer overflows, failure to check input data) related to known poor coding practices.
  • Common Vulnerabilities and Exposures (CVE) program provides a list of vulnerabilities containing an identification number, description, and one or more public references to a known issue in production software application 

Outlining strategy roadmap

  • Framework and maturity models, including SAMM and BSIMM, can assist organizations of any size with their formulation and implementation of a strategy for software security.
  • SAMM is an open framework intended to assist organizations of any size with their formulation and implementation of a strategy for software security.
  • SAMM Maturity levels:
    • 0—Implicit starting point representing unfulfilled security practices
    • 1—Initial understanding and ad hoc provision of security practice
    • 2—Increased efficiency and/or effectiveness of the security practice
    • 3—Comprehensive mastery of the security practice at scale
  • DevSecOps Maturity Model (DSOMM) is a recent OWASP project to mature software development operations and ensure operational software security is sufficiently matured.
  • DSOMM levels of maturation:
    • Level 1: Basic understanding of security practices
    • Level 2: Adoption of basic security practices
    • Level 3: High adoption of security practices
    • Level 4: Very-high adoption of security practices
    • Level 5: Advanced deployment of security practices at scale
  • Cybersecurity Maturity Model Certification (CMMC) is a program developed by the U.S. DoD that is designed to enforce protection and sensitive unclassified information that is shared by the Department with its contractors and subcontractors. 

Defining and developing security documentation 

  • A formal System Security Plan (SSP) provides an overview of the security requirements for an information system and describes the security controls that are either already in place or are planned to meet those requirements.
  • Documentation that supports SSP includes:
    • Configuration Management (CM) Plan 
    • Business Continuity Plan (BCP) including a Business Impact Analysis (BIA)
    • Continuity of Operations (COOP) Plan 
    • Continuous Monitoring Plan 
    • Security Awareness, Training, and Education Plan 
    • Incident Response (IR) and Disaster Recovery (DR) Plan
    • Privacy Impact Assessment (PIA) 

Defining security metrics

  • Key performance indicators (KPI) are metrics used by organizations to measure progress towards an organizational goal or objective.
  • Key risk indicators (KRI) are metrics that focus on monitoring risk using predefined thresholds for alerts.
  • Characteristics of good metrics include:
    • They are consistent. There should be no significant deviation between each measurement.
    • They are contextually specific. The metrics should be expressed in more than one unit of measurement.
    • They are inexpensive. The metrics should be generated automatically instead of manually.
    • They are objective. The findings should be indicative of the actual state of security in the software or system, and not the opinion of the person conducting the measurements or the stakeholder.
    • They are quantitative. The metrics should be precise and expressed in terms of cardinal numbers or as a percentage.
  • Common Weakness Scoring System (CWSS) describes poor coding practices (before production) within a range of 0 to 100.
  • Common Vulnerability Scoring System (CVSS) scores help organizations understand software vulnerabilities and prioritize risk mitigating actions.
  • The Software Assurance Metrics and Tool Evaluation (SAMATE) project seeks to standardize software security metrics reporting for various categories of software security assessment tools. 

Decommissioning applications 

  • When decommissioning software, an organization may be compelled to comply with local, federal, or industry compliance.
  • Decommissioning software may also have operating system, firmware, hardware, and data disposal considerations.
  • The first requirement in disposal of software and its related data and documentation is to establish an end-of-life (EOL) policy.
  • Data archiving is the process of identifying and moving inactive data from current production systems and into specialized long-term archival storage systems.
  • Data Disposition == disposal.
  • Decommissioning is the last step in the system life cycle.

Creating security reporting mechanisms 

  • A multitiered security reporting structure of three tiers: organizational, business process, and information systems is common within organizations.
    • The organizational tier provides the strategy and context for the entire organization, which includes lower-level reports. This level of the organization includes stakeholders such as the C-suite and the board of directors, as reporting must provide information that can be actioned at a high-level.
    • The next tier down, business process, might be synonymous within some organizations with a department (e.g., IT department, security operations center). At this tier, reporting is consolidated from each system, software, or network appliance into dashboards and other visual methods that allow for situational awareness within the mission or operation of the organization
    • At the lowest tier, we have information systems—end points. Reporting at this level is detailed and quantitative, with system logs, network logs, configuration baselines, and CVEs for software installed on hardware within the organization.
  • DevSecOps Security Reporting:
    • Plan: Opportunities and risks can be captured and recorded within the risk register. If using Agile, then the backlog could provide context for reporting.
    • Code/Build/Test: Some Integrated development environments (IDE) can provide developers trustworthiness of third-party libraries or linter suggestions based on language style guides. Often before source code can be committed to a repository it must pass a gated check-in where unit, integration, and/or static analysis tests review the source code for defects.
    • Release/Deploy/Operate/Monitor: Once the source is branched into an official release there will be a final statement regarding the software risks and mitigations. The organization or authorizing official will use a document such as a system authorization package, sometimes referred to as SAP, to decide whether the software system is authorized to operate.
  • Feedback loops (e.g., lesson learned, retrospectives) in reporting are integrally important to ensure stability and improvement over time within development processes. 

Incorporating integrated risk management methods

  • Organizations use risk assessment, the first step in the risk management methodology, to determine the extent of the potential threat, vulnerabilities, and the risk associated with an information technology (IT) system.
  • The output of this process helps identify appropriate controls for reducing or eliminating risk during the risk mitigation process, the second step of risk management.
  • The four primary risk strategies:
    • Mitigate: Controls (safeguards and countermeasures) can be implemented to reduce or mitigate risk. Given time and resource availability, this is the best course of action when addressing risk.
    • Accept: When the cost of implementing controls outweighs the risk itself, risk can be accepted if the acceptance is properly documented and formally performed. Risk acceptance with an exception to policy may be necessary when the software has unfixable vulnerabilities due to some legacy components in the software that are no longer supported. Risk acceptance must be performed only after the residual risk is known and there are contingency plans in place to address the risk as soon as possible.
    • Transfer: When the cost of implementing controls outweighs the risk itself, risk can also be transferred by buying insurance, but this is not common. Another risk transference mechanism that is prevalent in the world of software is the use of disclaimers (“as is” clauses) and end-user licensing agreements (EULAs), which are instruments that can be used to transfer the risk to the end user.
    • Avoid: Risk can be avoided by discontinuing the use of the vulnerable software and/or replacing or upgrading it with a more secure version.
  • Avoiding risk is different from allowing risk to be ignored (left unhandled)—the latter is never a good practice.
  • Residual risk is a reference to the remaining, potential risk after all IT security measures are applied. Risk can never be zero.
  • Risk monitoring enables organizations to verify compliance, determine the ongoing effectiveness of risk response measures, and identify risk-impacting changes to organizational information systems and environments of operation. 

Implementing secure operation practices 

  • Virtualization is a methodology for emulation or abstraction of hardware resources that enables complete execution stacks, including software applications, to run on it.
  • There are two types of hypervisors:
    • Type I hypervisor runs directly on computing hardware.
    • Type II relies on software to make a virtual layer.  
  • Containerization is a software process that bundles an application with minimal files and libraries it needs to run on any infrastructure.
  • Information security continuous monitoring (ISCM) describes maintaining ongoing awareness of information security, vulnerabilities, and threats to support organizational risk management decisions.
