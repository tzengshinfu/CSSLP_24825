Security Documentation (2.4)

This section will examine the topic of security documentation from two different perspectives. Both are equally important. Generally, at an overarching level there is a security plan referred to as the System Security Plan (SSP). Simply put, the SSP formally documents the protection of a system. There are also lower-level, securityrelevant documents to be considered during each phase of the development life cycle. Both are briefly covered here.

Objectives

  • Interpret the System Security Plan.
  • Recognize security-relevant documentation.

Overview

There are many ways to increase the security of software in development and the organizational development process. One security practitioner might perform a task completely different from one another, given the same business goal and no direction. Successful security documentation provides a flexible approach using guidelines while ensuring consistent results by leveraging procedures. Policies ensure that security team members and the business at large are uniform in approach to implementing and maintaining software security throughout the life cycle.

SSP

A formal provides an overview of the security requirements for an information system and describes the security controls that are either already in place or are planned to meet those requirements.

Security documentation for a software system must accurately capture/reflect security-relevant details on activities throughout various phases of the life cycle. Where and how these aspects and details are documented will be determined by different factors such as the prescribed formality in the adopted methodology (i.e., capturing security requirements in a “security requirement specification document” versus “inclusion of security stories in a sprint” as part of the sprint planning). NIST SP 800-18—Guide for Developing Security Plans for Federal Information Systems notes that “the objective of system security planning is to improve protection of information system resources. All federal systems have some level of sensitivity and require protection as part of good management practice. The protection of a system must be documented in a system security plan.”14

According to NIST SP 800-64—Security Considerations in the System Development Life Cycle, while the primary security documentation is the System Security Plan (SSP), other documentation supporting it may include:

  • including a Business Impact Analysis (BIA)
  • Security Awareness, Training, and Education Plan
  •  and Disaster Recovery (DR) Plan

System Development Documentation

Software security documentation should be created and updated throughout the SDLC. The security-relevant artifacts that are created in one phase of the life cycle may be needed and used in subsequent phases.

  • Requirements phase. During this phase, software security documentation could include the documentation for functional and nonfunctional security requirements and misuse/abuse cases, or any subject-object matrix used to generate the misuse/abuse cases. Misuse/abuse cases will be discussed in detail in future domains.
  • Design phase. During this phase, software security documentation could include the documentation of the threat model. As part of creating a threat model, additional documentation is produced, including the most common usage scenarios, various user roles, dependencies, assumptions, and threat scenarios. This documentation can be useful for those who perform code reviews, and for those who create security test cases, among other possible uses in subsequent phases of the life cycle. Threat modeling will be discussed in detail in future domains.
  • Implementation, testing, and subsequent life cycle phases. During these phases, similar documentation should be expected. The simplest way to relate to these documentation needs is to consider the security activities in each phase. Some examples of security-relevant activities related to the implementation, testing, deployment, operations, and maintenance of software that may require documentation include:
    • Attack surface analysis
    • Static analysis of application code
    • Manual security code reviews
    • Penetration test
    • Secure distribution and deployment
    • Secure configuration
    • Continuous monitoring

Software Security Documentation

Defining Documentation

Organizations create policies, procedures, and guidelines while leveraging standards to inform business operations. Security-centric documentation, including software security, follows the same conventions. Sofware security policies describe the objectives and constraints for organizational software security. Software policies are high level and provide the “what” and “why,” while procedures deal step-by-step with the “how.” Software security policies are normally technology independent.

For example, an organizational software security policy might dictate that all software development efforts must include software assurance activities to ensure new code is relatively free from defects prior to being committed to the organization’s repository. The organization could have procedures that provide more task-specific support for software assurance policy goals. These procedures could describe the steps to execute static and dynamic analysis within the organization. The procedures for static and dynamic analysis might be based on organizational tools and guided by industry standards to describe defects such as MITRE CWE. If software assurance efforts were automated via gated check-in for the organizations DevSecOps pipeline, an alternative supporting procedure might describe the steps to update the analysis configuration files for the gated check-in instead.

Developing Documentation

Security documentation is typically developed first at a strategic enterprise level with policy. Policies will inform the standards to inherit, as well as which procedures must be created to ensure the organization’s operational and development security. Not every aspect of an organization can or should be described step-by-step. Security guidelines can help bridge the gap between strategic policy goals and the direct tasking offered by procedures. When higher-level documentation such as policies change, it also drives change to lower-level procedures.
