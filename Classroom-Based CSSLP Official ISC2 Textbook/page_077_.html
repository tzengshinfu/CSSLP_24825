Runtime Protection (7.11)

Tools and technologies can afford applications protection against attacks that exploit common web application vulnerabilities. Operating systems offer protection mechanisms for applications at runtime.

Learning Objectives

  • Describe how controls contribute to protecting applications during runtime.

Overview

Controls are needed to protect applications at runtime. Controls include web application firewalls (WAFs), address space layout randomization (ASLR), and Data Execution Prevention (DEP). Web application firewalls can protect against many common web application attacks. Assigning random addresses to program components in the memory can help protect against certain types of attacks. DEP is a memory protection feature of the OS that can protect against malicious code.

Web Application Firewall (WAF)

The HTTP protocol used in web servers has been exploited by attackers in many ways. A web application firewall (WAF) can be an effective control in layered defenses. A WAF is an application firewall for HTTP applications. It applies a set of rules to an HTTP conversation. Generally, these rules cover common attacks such as Cross-site Scripting (XSS) and SQL Injection.28

WAFs operate based on a set of rules to filter out malicious traffic, in effect creating a shield between the web application and the internet by making clients pass through the WAF before they reach the server. WAFs can protect web applications against common attacks but should not be considered a substitute for fixing vulnerabilities in applications.

Whereas proxies generally protect clients, WAFs protect servers. A WAF is deployed to protect a specific web application or set of web applications. A WAF can be considered a reverse proxy. WAFs may come in the form of an appliance, server plugin, or filter, and may be customized to an application. It can take effort to perform this customization; this effort must be maintained as the application is modified.

Runtime Application Self-Protection (RASP)

Runtime Application Self-Protection (RASP) describes incorporating application security functionality within the software at runtime to reduce susceptibility to malicious events during execution. As the name suggests, RASP concentrates on application-level security instead of traditional network security controls. The application monitors itself during runtime and dynamically prevents exploits in real time.

RASP Implementation

WAF and IPS are more traditional security controls that monitor network traffic and user behavior to detect nefarious activity. Their scope is the network, not the application itself. RASP differs by moving the security mechanisms inside the application closer to the object requiring safeguards. RASP is specific to each application and achieves a level of accuracy and reduced latency in detection that is superior to more traditional solutions. RASP tools should provide two key features:

  • Threat intelligence. Not all vulnerabilities are known, as described definitionally by zero-day. Threat intelligence via RASP provides code execution level insights that assist security practitioners with addressing previously unknown security issues.
  • Application Protection. This provides a powerful medium for detecting known vulnerabilities and ameliorating them without disrupting business continuity through automatic runtime response.

RASP vs WAF

RASP and WAF both protect organizational network infrastructure from exploitation but differ in capabilities. A WAF protects infrastructure by analyzing and filtering network traffic between the application and the internet resource requestor; WAFs serve as a perimeter safeguard that blocks malicious traffic before it reaches the application. If the perimeter WAF is circumvented (e.g., alternate network path) then the protections will be ineffective.

However, RASP is path independent and instead relies upon contextual application data to prevent attacks. RASPs can prevent attacks that path-dependent security tools cannot. As infrastructure moves to the cloud and becomes more virtualized and containerized, the concept of network perimeter becomes harder to accept. One consequence of modern networking topologies includes the deteriorated effectiveness of firewalls both application and network specific. Security practitioners should consider WAF and RASP together for comprehensive and resilient cybersecurity.

Locality of Reference

The principle of locality of reference states that programs reuse data and instructions. The locality of reference principle has a corollary in the 90/10 rule, an empirical observation that a program spends 90% of its time in 10% of its code. Based on locality of reference, the instructions and data that a program has recently accessed will predict with reasonable accuracy what it will soon use.

Two different types of localities have been observed: temporal locality, which indicates that recently accessed items are likely to be accessed soon; and spatial locality, which indicates that items whose addresses are near one another tend to be referenced close together in time.

Address Space Layout Randomization (ASLR)

The address space layout randomization (ASLR) feature makes it difficult to create buffer-overflow attacks by randomizing the memory address for executables at system boot. This helps defeat a well-understood attack called “return-to-libc,” where exploit code attempts to call a system function such as the socket() function in wsock32.dll to open a socket, or LoadLibrary in kernel32.dll to load wsock32.dll in the first place. The job of ASLR is to move these function entry points around in memory, so they are in unpredictable locations. A DLL or EXE could be loaded into any of 256 locations, which means an attacker has a 1/256 chance of getting the address right. In short, this makes it harder for exploits to work correctly.

It should be noted that exploit techniques in the presence of ASLR have been introduced (e.g., return 0 programming), hence, ASLR should only be considered only as part of layered defenses. 

Data Execution Prevention (DEP)

DEP is a set of hardware and software technologies that perform additional checks on memory to prevent malicious code from running on a system. The primary benefit of DEP is to prevent code execution from data pages. Typically, code is not executed from the default heap and the stack. Hardware-enforced DEP detects code that is running from these locations and raises an exception when execution occurs. Software-enforced DEP can help prevent malicious code from exploiting exception-handling mechanisms.29

Security practitioners should understand the following concepts about DEP:

  • DEP is enforced by both hardware and software.
  • The Unix and Linux equivalent of Windows DEP is Executable Space Protection (ESP).
  • Exploit techniques for DEP are similar to ASLR.
