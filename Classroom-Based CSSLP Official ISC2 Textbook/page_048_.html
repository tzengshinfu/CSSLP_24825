Security Risks (5.4)

Organizations operate under a resource-constrained equation. There is rarely enough time, money, or business justification for perpetuating the prevention of every software defect. Organizations must be strategic in the pursuit of fixing or accepting technical and security debt (i.e., risks).

Objectives

  • Describe the process for defining a security strategy that includes considerations for software risks.

Overview

Every organization has a different risk appetite, tolerance, and landscape that make the tasks of identifying and ameliorating risks unique to each organization. This concept remains true when we focus on more specific areas of concentration such as secure software development. For example, software development projects will accept a certain level of risk by configuring DevSecOps pipelines to accept certain categories of issues while rejecting others. Perhaps a SAST (i.e., static analysis at gated check-in) tool is configured to detect only critical defects while potential moderate or warning-level findings are suppressed. In this case, the organization accepts any technical debt in the form of potential security defects within the source code by refraining from a more comprehensive scan. Analysis tools can produce many potential findings that may include many false positives (i.e., falsely reported issues) which could be prohibitively expensive to review in entirety. How do organizations cope with resource constraints?

Software Security Strategy Flow Down

Organizations must define a security strategy that includes considerations for software risks. The organizational strategy should capture the risk appetite and tolerance of the organization. There are several key software documents that, when drafted correctly, incorporate considerations for security based on the software engineering life cycle. Organizations flow strategic security to policies as shown above. The policies are then enforced using organizationally defined procedures. Having policy and procedure document artifacts makes it possible for the organization to define and conduct auditing and monitoring to ensure compliance. This describes how security strategy results in guidance at all levels for secure software activities throughout the life cycle, but it does not address creating a strategy specifically.

Creating a Software Security Strategy

Before an organization can mitigate software risks it must identify reasonable threats and important infrastructure. Not all threat sources possess the same capability to deliver negative impacts. Likewise, not all organizational assets are of the same importance. The standard risk approach starting with asset classification and protection needs can be applied to software development or even production software infrastructure.

First, organizations should define the critical software functions or components of the system when confidentiality, integrity, or availability are degraded, which result in unacceptable disruptions to the continuity of software operations. This exercise establishes priorities for which components may require a higher level of development scrutiny—prioritization by component criticality. Further, organizations may use threat intelligence information and a framework like MITRE ATT&CK to identify which adversarial events may be of higher likelihood of occurrence or greater impact to an organization—prioritization by adversarial forces. Prioritization may also be based on security budget or resources. If an organization has limited resources, it may only conduct static code analysis for critical defects while utilizing peer code reviews. Each software has different security and safety requirements based on the use case and operating environment. You would like the firmware on the airplane you’re traveling in to work every time. If you cannot log in to pay your bills today, you might not mind waiting until service is restored tomorrow.

Each organization’s decision to manage software risk is anchored on the aggregation of priorities as related to risks with the goal of reducing risk to an acceptable level for production software environments. Strategies should facilitate addressing defects in the highest criticality components starting with the highest impact items first to maximize value.
