Summary

Overall, security practitioners reviewing CSSLP Domain 5 should understand that the implementation phase represents the upfront opportunity to build secure, cyber-resilient software by incorporating pre-established architecture requirements and constraints into a tangible system. Software and system risk management and sustainment activities are less costly when identified earlier in the life cycle. Organizations that develop software should establish and enforce secure coding standards that include concepts such as robust input validation, output encoding, strong authentication mechanisms, session management, encryption techniques, and exception management. Secure coding standards often aim to minimize software defects, also known as weaknesses. The software implementation phase represents when software code is created.

Adhering to relevant secure coding practices 

  • SEI CERT produces the following coding standards for specific languages including C, C++, Perl, and Java.
  • In multithreaded environments, implementations must be thread-safe, that is, they must assure that accessing the same resource by various threads does not expose erroneous behavior or produce unpredictable results.
  • Input validation is one of the most fundamental and most critical mitigation strategies relevant to most common web application vulnerabilities (e.g., injection vulnerability, buffer overflow vulnerability).
  • Filtering is a form of validation.
    • An allow list (i.e., whitelist), is a list of “known good inputs.”
    • A block list (i.e., blacklist), is a list of “known bad inputs.”
  • Output sanitization means sanitizing the data you may already have and securing it prior to rendering it for the end user.
  • Proper error and exception handling must resolve anomalies, faults, and errors in ways that do not leave the software, its resources, data, or environment in a vulnerable state.
  • A type error is incorrect or undesirable program behavior caused by a discrepancy between different data types and includes considerations for constants, variables, methods, and functions. 

Analyzing code for security risks

  • Software assurance, commonly abbreviated as SwA, is the level of confidence that software is free from vulnerabilities, either intentionally designed into the software or accidentally inserted at any time during its life cycle, and that the software functions as intended by the purchaser or user.
  • Architectural Analysis refers to the system architecture, requirements, and design that comprise an application development life cycle.
    • Configuration Analysis. Software components can be configured in an insecure way, so it is important to monitor the configuration regularly.
    • Component Decomposition. Extracts dependencies from even cryptic pieces of software.
    • Software Supply Chain Analysis. Software Supply Chain Analysis determines the lineage of a software product.
    • Software Assurance Cases. An assurance case is a practical alternative to an overwhelming list or other unstructured methods for recording what was done or not done in a way that can lead to greater confidence in the result.
  • Dynamic application security testing (DAST) tools analyze applications in their dynamic running state during testing or operational phases. They simulate attacks against an application (typically web-enabled applications, services, and APIs), analyze the application’s reactions, and determine whether it is vulnerable.
    • Specialized Application Testing. Many applications provide specialized tools for penetration testing and analysis.
    • Dynamic Binary Analysis. Scanning a binary for weaknesses by executing the binary.
    • Dynamic Source Analysis. Some tools interpret source code for analysis or analyze source code while performing a dynamic analysis of an application.
  • Static application security testing (SAST) tools analyze an application source for security weaknesses, typically at the programming and/or testing software life cycle (SLC) phases.
    • Static Binary Analysis. Static binary analysis operates on a binary without utilizing the application’s source code.
    • Static Source Analysis. Static source analysis scans software code for weaknesses.
  • Interactive application security testing (IAST) tools combine elements of DAST with instrumentation of the application under test. 

Implementing security controls 

  • Controls == safeguards == countermeasures whether proactive or reactive are used to mitigate risks to assets.
  • Prevention, detection, and correction of vulnerabilities require implementation of various mitigation measures.
  • To prevent injection, keep data separate from commands and queries.
  • To prevent broken authentication, implement multifactor authentication, check for weak passwords, limit failed attempts, rely on client and server-side validation.
  • To prevent broken access, deny access by default, disable web server directory listing, rate limit APIs, and implement a single access mechanism scheme for the enterprise.
  • To prevent sensitive data exposure, label, classify, and implement protections accordingly. Leverage encryption. Discard unneeded sensitive data. Hash stored passwords.
  • To prevent XXE, ese fewer complex data formats such as JSON and avoid serialization of sensitive data.
  • To prevent XSS, separate untrusted data from active browser content.
  • To prevent insecure deserialization, thought during architecture and design are required so that deserialization of untrusted sources will not be required. Only permitting primitive types for example. 

Evaluating and integrating components 

  • Obtaining software components from unverified sources, acquiring components over insecure links, or even using older versions of components that are not maintained (unpatched) are a few other issues among a much longer list of issues pertaining to the use of third-party code or libraries.
  • Software Composition Analysis (SCA) tools is to gain visibility into the open-source inventory (open-source components used in a software product) and help automate the process for open-source management.

Applying security during the build process

  • Security controls and checks should be factored into an automated build process wherever possible.
  • Compiler switches and warning posture may help with the enhancement of security.
  • Code obfuscation in binary format is about introducing another layer of defense against reverse engineering to protect intellectual properties.
  • Code signing is a reference to digitally signing executable code to provide assurance of the source integrity and provide confirmation of the software author.
  • Code signing and obfuscation are examples of anti-tamper mechanisms.
