Evaluation and Integration of Components (5.5)

Modern applications are typically built with dependance on third-party components and libraries. Software security practitioners must understand the great and real risk of using these libraries with known vulnerabilities. The sole task of identifying these commercial or open-source components could present a great challenge. Further, these libraries may be updated and patched over time, and this highlights the need for the continuity of the risk assessment processes using automated and manual processes.

Objectives

  • Identify the risks associated with using third-party and open-source components and libraries.
  • Describe the various integration categories and their relevance to software security.

Overview

Development teams frequently rely on third-party components and APIs for specific functions within their overall software products. These components provide a plug-and-play opportunity for development teams and eliminate the need to reinvent and develop the capabilities from scratch. For all the benefits of third-party components, there are also concerns and drawbacks. These include relinquishing control over the development processes of the code and relying on third parties to update and maintain the security of their components.

Securely Reuse Third-Party Code or Libraries

Third-party code or libraries fall in the category of software whose security risk must be managed. Using components with known vulnerabilities is one part of the problem. Obtaining software components from unverified sources, acquiring components over insecure links, or even using older versions of components that are not maintained (unpatched) are a few other issues among a much longer list of issues pertaining to the use of third-party code or libraries. Static analysis tools that are traditionally used with software developed in-house, may also be used with third-party components.

Third-Party Code

The increased standardization of software functions, protocols, and interfaces has enabled a large amount of code that would have been custom developed in the past to instead be written once by someone and reused by many. Frequently, the code is not written by the developers of the system or application in which the code will be used, or even by developers in the same organization.

The practice of software reuse, which began as a means of helping developers leverage their own well-engineered custom code and that of other developers in their organization, has long since evolved into use of third-party and open-source libraries of interfaces, protocols, and functions; software development kits (SDKs); wrappers; application frameworks; and a variety of other third-party software components—the pedigree and provenance of which is often obscure and sometimes unknowable.

A library or SDK that is commercially sold or extremely popular may still contain vulnerabilities. The original developers could have been careless or malicious, or it could have been dangerously tampered with at some point between its original release and its acquisition by the development team that plans to reuse it.

Third-Party and Open-Source Components—Risk Factors

While it is certainly important to consider the possibility and the risk of malicious code being inserted in software at some point (particularly when software is developed in a geographically distributed supply chain), security flaws and vulnerabilities introduced in software inadvertently due to insecure design and coding practices or errors and mistakes represent a higher risk. Risk factors associated with third-party and open-source components include:

  • The absence of an accurate inventory of all third-party/open-source components
  • Outdated technology
  • Unsupported components (component age)
  • Repository trust
  • License

Open-Source Policy

According to OWASP40, open-source policies provide guidance and governance to organizations looking to reduce third-party and open-source risk. Policies typically include:

  • Restrictions on component age
  • Restrictions on outdated and EOL/EOS components
  • Prohibition of components with known vulnerabilities
  • Restrictions on public repository usage
  • Restrictions on acceptable licenses
  • Component update requirements
  • Block list of prohibited components and versions
  • Acceptable community contribution guidelines

While the open-source policy is usually filled with restrictions, it provides an organization’s security, development, and legal teams an opportunity to create solutions for healthy open-source usage.

Software Composition Analysis (SCA)

Third-party/open-source components are an essential part of contemporary software development. Critical vulnerabilities identified in third-party/open-source components, libraries, and frameworks highlight the need for Software Composition Analysis (SCA). At the time of this writing, the primary objective of using SCA tools is to gain visibility into the open-source inventory (open-source components used in a software product) and help automate the process for open-source management. SCA tools can help with both security as well as license management.

Software component integration has been an aspect of system designs from the early days of software to the present day. Applications may be constructed using a combination of components developed in-house and components that are developed by third parties. Components may be designed and developed by different groups using different specifications and assumptions, but for the system to function properly, these components all must interoperate properly and securely.

Web-based APIs have been around since the 2000s but seem to have gained widespread acceptance and usage over the past decade.

API Integration

It is common for organizations to build applications that integrate with third-party APIs. Developers now rely on Software Development Kits (SDKs) and APIs to add capabilities to their applications that otherwise would have been difficult or impossible to achieve.
Among the capabilities that have been made possible through integration with third parties are the following:

  • Maps
  • Machine learning
  • Email delivery
  • Messaging (e.g., SMS)

API integration has brought an unprecedented opportunity for application developers to build intelligent and powerful applications quickly. To make this integration possible, API developers frequently provide libraries to support various programming environments including Java, C#, PHP, and Python, among others.

API Comparison

SOAP, REST, and GraphQL may be used to create APIs, but there are some major differences among them.

Briefly, two basic functions, GET and POST, are used to retrieve, modify, and add data from or to the server with SOAP. With REST, the state of sources is changed by making requests to Uniform Resource Identifiers (URI). With GraphQL, two types of requests are leveraged to query and mutate data.

SOAP involves a large amount of data transfer with request and the response side, resulting in slowdown of communication and demand for more resources. In comparison, REST is more efficient than SOAP. GraphQL addressed the problem of over/under fetching associated with other approaches by allowing the specification of exactly what needs to be fetched.

Here, a basic distinction between different APIs has been made, and there is much more beyond what has been covered. But essentially, when selecting an API, developers need to understand and address security requirements.

API Management

APIs may be created for internal use only (private), or they may be intended to be used by specific partners; as a third option they may be meant to allow third-party application developers to interact with them (public). HTTP-based APIs are used frequently. APIs must be designed, published, and documented in a secure manner or environment. Organizations that develop APIs should consider API management solutions or platforms for this purpose.

API management refers to the processes for distributing, controlling, and analyzing the APIs that connect applications and data across the enterprise and across clouds. The goal of API management is to allow organizations that create APIs or use others’ APIs to monitor activity and ensure the needs of the developers and applications using the API are being met.41

API Security

Security of APIs was discussed in previous domains and OWASP API Security Top 10 was pointed out. Given the prevalence of Representational State Transfer (REST), a few key topics with respect to security are highlighted below.

  • Secure REST services must only provide HTTPS endpoints.
  • Nonpublic REST services must perform access control at each API endpoint.
  • If using JSON Web Tokens (JWT), integrity of the JWT must be protected.
  • HTTP methods must be restricted.
  • Request content types must be validated.

Continuous Integration

Given the current industry attention to Continuous Integration (CI), and the relevance of integration to this module, a brief overview of continuous integration is presented below.

When software developers work in isolation, the task of integrating their code with other team members’ code may present a challenge that can lead to merge conflicts, hard-to-find bugs, duplicated efforts, and other challenges. This issue may subsequently slow down the processes of software release.

Continuous integration can offer advantages over a traditional end-of-life-cycle approach, including:

  • More agility to respond to business requirements as the tempo of software change and release increases
  • Early detection of issues
  • Reduction of the integration risks due to a decreased complexity
  • Continuous feedback from software users to software development teams due to increased and incremental releases of the software

Continuous integration may rely on the integration of code that is checked into shared code repositories on a frequent basis. By automating the build processes, various security checks and tests can be orchestrated using commercial and open-source tools and technologies that have been introduced for this purpose.

Various tools and technologies for orchestrating the chain of actions needed for continuous integration have been introduced. Jenkins is a good example of open-source CI server used for this purpose. Regardless of the tools or technologies utilized, best practices for securing the CI/CD pipelines must be followed.

Systems-of-Systems Integration

Organizations in military, civilian government, and commercial domains need to engineer systems of systems rather than stand-alone systems to meet challenges such as:

  • Collaboration across independently funded and managed organizations
  • Migration to a service-oriented environment
  • Testing and compliance verification processes for systems of systems
  • Interface particulars of systems to be integrated
  • The degree of uncertainty
  • Change in constituent systems and interfaces
  • Assurance and dependability/supportability requirements

Ensure enough robustness and regression testing and analysis activities are incorporated into the development phase, including those that exercise quality related scenarios. Require a risk-based verification plan that focuses on robustness of critical system services in the face of uncertainty, complexity, and change.
