15. What is the most important aspect of code signing? (D1.1, D5.6, D7.3) (L5.14) 

A. Ensuring that the code is in the correct format for the signing process
B. Ensuring that the correct public keys are available to sign the code
marked correct C. Ensuring the origin and integrity of software 
D. Ensuring hashing algorithms can perform both encryption and decryption
Feedback

Your answer is correct.

The correct answer is C. Code signing is the process of digitally signing executables and scripts to confirm the software author and guarantee that the code has not been altered or corrupted since it was signed. The process involves creating a digest by hashing the file's contents and encrypting it with the private key of the software publisher. This is done to validate authenticity and integrity. This requires the entity that signs to code to assure proper protection of the code signing key.

Incorrect answer feedback: Code signing is not about confidentiality or public key infrastructure (PKI).

18. What is the primary purpose of a separation kernel in the context of security? (D5.1) (L5.5)

A. To handle complex operating system responsibilities
B. To provide virtualization features for virtual machines
marked correct C. To ensure the correctness of the separation function 
D. To support device drivers and shell access
Feedback

Your answer is correct.

The correct answer is C. The primary purpose of a separation kernel is to ensure the correctness of the separation function. By factoring out the separation function from a complex operating system and creating a kernel solely focused on separation, it becomes possible to verify the correctness of the separation mechanism. This is important for maintaining the security and isolation of sensitive information in high-security systems.

Incorrect answer feedback: The kernel becomes simpler and more verifiable by focusing solely on separation. While separation kernels can utilize virtualization features to define fixed virtual machines (VMs), the primary purpose of a separation kernel is not to provide virtualization. Additionally, separation kernels typically eliminate or minimize functionalities like device drivers and shell access.

19. What is the difference between static and dynamic linking? (D5.6) (L5.13, L5.14)

A. Static linking involves combining source code files, while dynamic linking involves running tests on the compiled code.
marked correct B. Static linking links all library dependencies at compile time, while dynamic linking links library dependencies at runtime.
C. Static linking is used for large-scale software projects, while dynamic linking is suitable for small-scale projects.
D. Static linking is more vulnerable to attacks than dynamic linking due to its reliance on external libraries.
Feedback

Your answer is correct.

The correct answer is B. In static linking, all library dependencies are resolved and linked with the executable during the compile-time process. The resulting executable contains all the necessary code from the libraries. On the other hand, dynamic linking resolves library dependencies at runtime, allowing the executable to load and link with the required libraries when the program is launched.

Incorrect answer feedback: Static and dynamic linking are not related to the process of combining source code files or running tests. They specifically refer to how libraries are linked with the executable. The choice between static and dynamic linking is not determined by the scale of the software project. It depends on factors such as the specific requirements of the project and the desired flexibility in managing library dependencies. The vulnerability to attacks is not directly influenced by static or dynamic linking itself. It depends on other factors, such as the security of the linked libraries. Both static and dynamic linking have their own security considerations.

20. Which of the following is a prevention strategy for buffer overflow vulnerabilities? (D5.2) (L5.7)

A. Increasing the buffer size to accommodate more data
marked correct B. Avoiding the use of unsafe APIs 
C. Allowing direct memory manipulation through pointers
D. Encouraging unrestricted copying of buffers
Feedback

Your answer is correct.

The correct answer is B. One of the prevention strategies for buffer overflow vulnerabilities is avoiding the use of unsafe APIs. Unsafe APIs, such as "strcpy," can lead to buffer overflows if not used with proper buffer size checks. By using safer alternatives or implementing buffer size checks and validation, the risk of buffer overflow vulnerabilities can be mitigated.

Incorrect answer feedback: Increasing the buffer size may provide more space, but it does not address the core issue of buffer overflow vulnerabilities. Allowing direct memory manipulation through pointers can actually increase the risk of buffer overflow vulnerabilities. Finally, unrestricted copying of buffers is one of the main causes of buffer overflow vulnerabilities.
