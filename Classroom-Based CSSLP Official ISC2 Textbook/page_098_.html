Summary

Overall, security professionals reviewing CSSLP Domain 6 must understand that testing software involves the development of both functional and nonfunctional tests. Functional testing aims to assess the functionality of the software in respect to functional specifications. Nonfunctional testing aims to assess cyber resiliency, performance, safety, and useability. Software assurance is a process and specialization within cybersecurity that seeks to establish the level of confidence that a software is free from vulnerabilities, either intentionally designed into the software or accidentally inserted at any time during its life cycle, and that the software functions as intended by the purchaser or user.

Although software assurance is an integral part of secure software testing, it is important to understand that derivative artifacts from software engineering must also be inspected such as installation procedures, error messages, and release notes. Security testing must provide adequate coverage, at the appropriate life cycle phases. Security testing results must be consistent with organizationally defined levels of acceptable for error, accuracy, and precision to provide meaningful guidance.

Further, testing should be supported from the top down, starting with enterprise guidance in the form of security testing strategy. Organizational policies such as software development plans (SDP) and software test plans (STP) aim to achieve goals defined within the strategy. Test procedures, checklists, and automation enhance testing quality and consistency.

  • Black box testing
    • testing remains focused on inputs and outputs and what the software does
    • the tester has no knowledge of inner components/structures
  • Gray Box Testing is between black box and white box testing lies gray box testing, which uses limited knowledge of the program internals.
  • The threat modeling should include a logical and physical, end-to-end scenario, and when you are performing environment testing, all the components should be thoroughly tested.

Verifying and validate documentationÂ 

  • Software document verification is the process of ensuring the authenticity and integrity of documentation.
  • Software document validation is the process of ensuring the correctness, standards conformity, policy conformity, and consistency of documents.
  • To safeguard against issues related to manual document reviews, organizations typically implement policies, procedures, checklists, and guidelines to increase consistency and reduce efforts.
  • Automatic document checking can be faster, cheaper, and more accurate than manual methods, but it is generally cost prohibitive to automate everything.
  • Application Security Verification Standard (ASVS) defines three security verification levels with increasing rigor:
    • ASVS Level 1: the bare minimum that all applications should strive for.
    • ASVS Level 2: appropriate for applications that handle significant business-to-business transactions, including those that process healthcare information, implement business-critical or sensitive functions.
    • ASVS Level 3: reserved for applications that require significant levels of security verification, such as those that may be found within areas of military, health and safety, critical infrastructure, etc.Â

  • When security testing results reveal security flaws and vulnerabilities in software, the product team, in coordination with other stakeholders (e.g., business), needs to prioritize the remediation efforts.
  • When testing reports an issue that is really a non-issue, the result is said to be a false positive, which is also referred to as a Type I error.
  • Type II errors, false negatives, occur when a tool fails to discover and report real issues.

Classifying and track security errors

  • The Common Vulnerability Scoring System (CVSS) is a published standard used by organizations worldwide to capture the principal characteristics of a vulnerability.
  • CVSS considers the following to calculate a score from 0 to 10:
    • Base metrics for qualities intrinsic to a vulnerability.
    • Temporal metrics for characteristics that evolve over the lifetime of the vulnerability.
    • Environmental metrics for vulnerabilities that depend on a particular implementation or environment.

Securing test data

  • If sensitive data is extracted from production databases and moved or used as test data, the security of the sensitive data may be compromised. This data must be sanitized by whatever means possible before it can be used.
    • Data masking and data obfuscation are the processes of hiding, replacing, or omitting sensitive information from a specific data set.
    • Anonymization is the process of removing identifiers to prevent identification of individuals or sensitive information.
    • Tokenization, like anonymization, aims to prevent identification of individuals or leakage of sensitive information but involves swapping sensitive data with a generated identifier (i.e., token) which cannot be reserved to access the sensitive data.
    • Sanitization is a process to render access to data on the media infeasible for a given level of effort such as clear, purge, or destroy.
  • Relational database management systems (RDBMS) must provide methods for preserving the consistency and integrity of data in the database.Â
