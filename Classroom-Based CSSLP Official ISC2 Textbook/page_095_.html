Summary

Overall, security practitioners reviewing CSSLP Domain 3 need to understand the various laws, regulations, and industry standards that impact security and privacy requirements for secure software development. Good software security begins with a good organizational structure that supports, defines, and addresses security needs through internal policies and procedures. This organizational structure in tandem with software project stakeholders should flow into defining software security requirements. Organizations must capture and specify software security requirements in tandem with overall requirements for the software product.

Compliance is a significant source of software security requirements. Organizations must establish policies and procedures to classify information based on levels of sensitivity, value and mission criticality. Sensitive data, whether structured or unstructured, must be protected throughout the software and data life cycle. Good security requirements, whether contractually to a supplier or internal to a development team, establish upfront expectations which when successful drafted may help ameliorate the need for excessive production security safeguards implemented later in the life cycle.

Defining software security requirements

  • A functional requirement specifies something the system must do or describes a necessary product feature.
  • Functional software security requirements describe the needed security functionality and features of a software product.
  • Non-functional requirements describe how well a system will operate, including security, performance, and safety specific capabilities and constraints
  • Use cases describe user-focused scenarios that represent what a system does when used in the ways for which it was designed.
  • Misuse (abuse) cases describe how users misuse or exploit the weaknesses of controls in software features to attack an application.
  • Technical debt is the consequences of software development actions that intentionally or unintentionally prioritize client value and project constraints

Identifying compliance requirements 

  • A policy is important to establish both internally and externally what an organization’s information security architecture position.
  • Policies are supported by standards, baselines, procedures, and guidelines:
    • Standards: Standards are mandatory activities, actions, rules, or regulations designed to provide policies with the support structure and specific direction they require to be meaningful and effective.
    • Baselines: Baselines define a set of basic security objectives that must be met by any given service or system.
    • Procedures: Procedures state the step-by-step specifics of how the policy and its supporting standards and guidelines will be implemented in an operating environment.
    • Guidelines: Guidelines are nonmandatory best practices, often useful as source material for creating standards and procedures 

Identifying data classification requirements 

  • Organizational data should be properly labeled, and a classification system implemented before data can be protected.
  • The data classification process requires a consistent approach to categorizing data based on criteria that are well thought out, specific, and predefined.
  • Data Governance is a system of decision rights and accountabilities for information-related processes, executed according to agreed-upon models which describe who can take what actions with what information, and when, under what circumstances, using what methods.
    • The Data owners are typically business groups within the enterprise who are responsible for any negligent acts that result in corruption or disclosure of the data.
    • The data custodian is given the responsibility for the maintenance and protection of the data.
    • The data subject is the individual that personal data is about.
  • Data takes one of three general forms:
    • Structured data types that are centrally managed and often accessed and managed using a database or repository like Structured Query Language (SQL).
    • Unstructured data does not have a predefined data model or is not organized in a predefined manner and is, therefore, difficult to process or analyze using conventional tools and methods.
    • Semistructured data relies on metadata (e.g., JSON, CSV, XML) to aid data cataloging and analysis. 

Identifying privacy requirements 

  • Data protection and privacy laws and regulations provide protection against misuse and disclosure of personally identifiable information (PII), personal healthcare information (PHI), and individuals’ financial information.
  • The adverse impact of privacy events is typically in two areas: loss of public trust and legal liability. 

Defining data access provisioning

  • Subjects—employees and stakeholders—are onboarded and offboarded and therefore have life cycle considerations.
  • Objects—the data and resources they require access to—are created, managed through the life cycle, and eventually disposed. 

Developing misuse and abuse cases

  • To contrast use cases, misuse/abuse cases incapsulate thinking with the perspective of malicious subjects aiming to inflict damage rather than end-user activities.
  • Attack patterns help organizations understand potential abuse and misuse.
  • Common Attack Pattern Enumeration and Classification (CAPEC) is a good example of a publicly available catalog of common attack patterns, where patterns are presented by the mechanism or the domain of attacks.
  • MITRE ATT&CK is a well-known knowledge base of adversarial techniques—a breakdown and classification of offensively oriented actions that can be used against platforms.

Developing security requirement traceability matrix

  • Security Requirements Traceability Matrix (SRTM) as a matrix documenting the system’s agreed upon security requirements derived from all sources, the security features’ implementation details and schedule, and the resources required for assessment.
  • The SRTM can be used for any type of project and allow security requirements and security tests to be easily traced back to one another.  
  • There are three types of SRTM:
    • Forward traceability. Maps requirements to test cases.
    • Backward or reverse traceability. Maps test cases to requirements.
    • Bidirectional traceability. Maps requirements to test cases and vice versa.
