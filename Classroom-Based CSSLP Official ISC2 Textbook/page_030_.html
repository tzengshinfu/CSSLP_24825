Domain Review

Good software security begins with a good organizational structure that supports, defines, and addresses security needs through internal policies and procedures. This organizational structure in tandem with software project stakeholders should flow into defining software security requirements. Organizations must capture and specify software security requirements in tandem with overall requirements for the software product.

Beyond functional requirements for software systems, organizations need to stay abreast of applicable laws, regulations, and industry standards based on consumer and data jurisdictions. Compliance is a significant source of software security requirements. Now more than ever consumer rights and privacy are a rapidly developing frontier. Organizations must establish policies and procedures to classify information based on levels of sensitivity, value and mission criticality. Sensitive data, whether structured or unstructured, must be protected throughout the software and data life cycle.

Good security requirements, whether contractually to a supplier or internal to a development team, establish upfront expectations which when successful drafted may help ameliorate the need for excessive production security safeguards implemented later in the life cycle.
