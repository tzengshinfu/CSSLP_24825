Domain Review

Software security testing ensures software is reasonably free from defects. Defects are represented throughout the life cycle as risks, weaknesses, or vulnerabilities, depending on system engineering and SDLC phases. Security testing must provide adequate coverageÂ at the appropriate life cycle phases. Further, security testing results must be consistent with organizationally defined levels of acceptability for error, accuracy, and precision to provide meaningful guidance.

Testing should be supported from the top down, starting with enterprise guidance in the form of security testing strategy. Organizational policies such as software development plans (SDP) and software test plans (STP) aim to achieve goals defined within the strategy. Test procedures, checklists, and automation enhance testing quality and consistency. Software testing should include more than source code. Artifacts such as documentation and logs should be included in test considerations.

Security practitioners should strive to facilitate the creation of trustworthy secure software systems. Trust is not granted based on individual judgment but is instead based on the demonstration of trustworthiness. A software system is deemed trustworthy when sufficient evidence exists to support claimed trustworthiness. Security testing demonstrates the ability and, therefore, the worthiness of a software entity to be resilient in the face of adversity. Through this relationship, we grasp the undeniable importance of security testing throughout the life cycle.
