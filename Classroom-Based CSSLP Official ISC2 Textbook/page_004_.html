Security Design Principles (1.2)

Secure design principles define practices that can guide decisions at the system and software architecture level. It is important for security practitioners to understand architecture and design principles are abstracted away from platform and programming language specifics. These principles lay the foundation for building and maintaining secure software.

Decisions are made to establish adequate security, which represents intentional trade-offs to reach a state of security that supports the business function of the system or software. Sometimes one principle is given more consideration than another based on how the organization defines the system as reasonably secure. According to NIST, secure systems typically exhibit the following characteristics:

  • It enables delivery of system capability despite all manner of adversity.
  • It constrains functions to desired behaviors based on system required capabilities.
  • Constraints are enforced via predefined rulesets to ensure only authorized interactions and operations are allowed.

Objectives

  • Explain how security methods mitigate vulnerabilities through access controls.
  • Describe the purpose and function of multiple layers of protection in software security.
  • Describe how security culture and practices impact data privacy and security.

Overview

Secure design principles define practices that can guide decisions at the architecture level regardless of the platform and regardless of the programming language. These principles include Least Privilege, Separation of Duties, and Defense in Depth, among many more.

Least Privilege

Least privilege is the concept of providing no more authorizations than necessary to perform the required functions. This also implies that if an operation requires elevated privileges, such privilege elevation should be limited to the shortest time required to complete that operation.

The principle of least privilege states that each component should be allocated sufficient privileges to accomplish its specified functions, but no more. This limits the scope of the component’s actions, which has two desirable effects: the security impact of a failure, corruption, or misuse of the component will have a minimized security impact; and the security analysis of the component will be simplified. Least privilege is a pervasive principle that is reflected in all aspects of the secure system design.

For example, if a middleware server only requires access to the network, read access to a database table, and the ability to write to a log, this describes all the permissions that should be granted. Under no circumstances should the middleware be granted administrative privileges.2

Run Time Privileges

Understand that violation of the least privilege principle at run time can lead to the magnification of inflicted harm if a software vulnerability is discovered and exploited. A real-life example would be how excessive app permissions at run time can cause many different issues related to security and privacy.

A classic example would be when the input validation mechanisms in software fail to prevent SQL injection attacks (i.e., due to ineffective regular expressions) and the vulnerability is successfully exploited. When this occurs, elevated privileges may lead to significantly more harmful payloads such as wiping out one’s data.

Another example is the early-release flashlight apps that were downloaded from app stores for smartphones and other mobile devices. These apps require access to one’s location data and the contacts on the mobile device. Of course, most security-conscious users were concerned with these required app permissions and the contents/resources that were accessed by these types of apps, but the apps were nevertheless installed and used. A clear message that should be conveyed is that addressing the least privilege principle is critical, but doing so may be a complex process and require considerable effort.

Zero Trust

NIST defines within SP 800-207 “Zero Trust Architecture” as a collection of concepts and ideas designed to minimize uncertainty in enforcing accurate, least privilege per-request access decisions in information systems and services in the face of a network viewed as compromised. ZT places requirements for continuous validation and authentication on all users (subjects) whether inside or outside of the organization, without respect to entry points, before authorizing resources such as data. Organizations implementing ZT must develop an implementation plan which includes the following technology agnostic tenants:

  1. Data sources and computing services are resources
  2. All Communication is secured
  3. Access to resources is granted individually on a per-session basis
  4. Access to resources is determined by dynamic policy
  5. Continuously monitor and measure the integrity and security posture of assets
  6. Authentication and authorization are dynamic and strictly enforced
  7. Collects adequate data to improve and maintain security posture

Beyond tenants of a ZT implementation plan, the organizational implementation typically includes three components: a policy engine, a policy administrator, and a policy enforcement point. The policy engine coordinates with the policy administrator component to action and log the approve-deny decisions originating from the policy administrator. The policy administrator brokers an access path from the policy enforcement point. This communication is often referred to in the context of a “control plane.” The policy enforcement point is responsible for initiating, terminating, and monitoring connections using a client-server access model. Simply put, security practitioners should understand that zero trust is a high-level strategy that once implemented should enforce the same level of mutual rigor for all subjects accessing objects within the organization.

Separation of Duties

Separation of duties is the concept that no subject should be given enough privileges to misuse the system on their own. This implies that critical functions must be divided among subjects to mitigate the risk of fraud.

For example, an administrator should be able to turn the system on or off, and set password policy, but shouldn’t be able to log on to the storefront as a super-privileged user, such as being able to ‘buy’ goods on behalf of other users.3

Applying proper separation of duties clearly requires subjects/actors performing the separated duties to be different entities, but there may be different levels of segregation involved. At the most basic level, the segregation may involve two different actors that are performing similar functions and duties (e.g., two actors from the same department). Taking this to the next level, segregation could involve different actors that perform dissimilar functions/duties (e.g., subjects from two different departments).

An example of this principle in the realm of secure software would be the considerations for the separation of the encryption process from the key management process.

Split Knowledge

The concept of split-knowledge can be explained by describing a process where a cryptographic key is split into n multiple key components, individually providing no knowledge of the original key, which can be subsequently combined to recreate the original cryptographic key.4

Secret Sharing

The concept of secret sharing is not new, but some interesting technologies and solutions have recently evolved around it. Algorithms have been introduced in cryptographic protocols that attempt to segment a secret (e.g., encryption key) and store it in a distributed way among many participants, thus making it infeasible for an attacker to recover the secret by compromising a single node.

Defense-in-Depth

Defense-in-depth involves applying multiple layers of protection where a subsequent layer will provide protection if a different layer is breached. This implies that controls in one layer could compensate for weaknesses of controls in other layers. It also implies that in situations where one control may be considered reasonable, addressing risks through multiple controls that approach risks in different fashions might be a more reasonable approach. Defense in depth describes security architectures constructed through the application of multiple mechanisms to create a series of barriers to prevent, delay, or deter an attack by an adversary.5

For example, a flawed administrative interface is unlikely to be vulnerable to anonymous attack if it correctly gates access to production management networks, checks for administrative user authorization, and logs all access.6

Layered Controls

To better appreciate the value of a layered approach to the positioning of controls, with the objective of creating strength and depth for mitigating the effects of a threat, a bit of history from the realm of physical security could be useful:

Defense-in-depth as a concept originated in military strategy to provide barriers to impede the progress of intruders from attaining their goals while monitoring their progress and developing and implementing responses to the incident to repel them. In the cybersecurity paradigm, Defense-in-depth correlates to detective and protective measures designed to impede the progress of a cyberintruder while enabling an organization to detect and respond to the intrusion with the goal of reducing and mitigating the consequences of a breach.7

Input Validation

One way to protect software against externally influenced (malicious) input is to prevent its entry. Input validation should be considered one of the most important strategies to protect against the most common vulnerabilities in Web applications. It will be examined in future domains in greater detail. For now, input validation should be recognized as an important part of your overall defenses in software.

Security Zones

The concept of security zones is relevant in many realms of security, including physical security, network security, and software security. The creation of effective layered defenses in all these realms requires a clear understanding of how various controls or pieces of technology can fit together. In theory, a system may have any number of security zones, and each zone may have any number of controls. How many zones and what controls are needed will depend on several factors, including the type of assets in each zone and the level of protection that the various assets require.

Diversity of Defense

During the discussion of defense-in-depth principles, we highlighted the importance of controls positioned at various layers of defense. Also important is the assurance that these controls are diverse in their capabilities, as each may play a specific role in the overall defenses of the system. Further, diversity also reduces the likelihood that a common flaw or vulnerability would be exhibited by all such controls.

Organizations may use diversity in several ways to enhance the resiliency of a system. Systems supporting software or contingency could be in geographically separate areas, which might insulate them from a natural disaster disrupting system operations. If a project is large and critical, an organization might ensure that any systems and software being supplied are from multiple vendors or leverage multiple technologies.

Resiliency

Software by its nature will fail at some point due to some reason. To be resilient, software needs to fail in a safe/secure manner. One requirement is that when a system fails, it should fail to a state where the security of the system and its data are not compromised.

A security principle that aims to maintaining confidentiality, integrity, and availability by defaulting to a secure state, rapid recovery of software resiliency upon design or implementation failure. In the context of software security, fail secure is commonly used interchangeably with fail safe, which comes from physical security terminology.8

Resiliency may also be undermined when a single point of failure or a single point of compromise in a system exists. Clearly, single points of failure are undesirable in any system with a goal of availability. Eliminating single points of failure is achieved by implementing redundancy and compensating for controls.

Economy of Mechanism

demonstrates the principle, “keep it simple, stupid.” Overly complex approaches will not necessarily enhance security as opposed to relatively straightforward and simple approaches. This also implies that the more complex the design of a system/software, the higher the likelihood that a given vulnerability may go unnoticed.

This concept of simplicity is echoed by the principle of Build Security In (BSI): One factor in evaluating a system's security is its complexity. If the design, implementation, or security mechanisms are highly complex, then the likelihood of security vulnerabilities increases. Subtle problems in complex systems may be difficult to find, especially in copious amounts of code. For instance, analyzing the source code that is responsible for the normal execution of a functionality can be a difficult task, but checking for alternate behaviors in the remaining code that can achieve the same functionality can be even more difficult.

An example presented by OWASP can help us better relate to this principle: “Although it might be fashionable to have a slew of singleton entity beans running on a separate middleware server, it is more secure and faster to simply use global variables with an appropriate mutex mechanism to protect against race conditions.9"

Password Vaults

Encrypted vaults in general aim to offer secure storage for secrets such as keys and passwords. For access management, especially privileged access, password vaults may be used. These vaults may be used by password managers.

Greater security is achieved principally through the capability of most password manager applications to generate unique, long, complex, easily changed passwords for all online accounts and the secure encrypted storage of those passwords either through a local or cloud-based vault.10

Resource Efficiency

Software will not function without dependencies. Software relies on underlying hardware resources such as storage mediums, CPUs, and RAM which must be allocated appropriately when developing source code. Further, applications must efficiently free—that is, deallocate—resources when execution of a task has completed. When resources are not managed properly due to poor coding practices, the implications extend beyond inefficiently run software. Poorly implemented and designed software can deprive other software running on the same resources (e.g., server, hypervisor) of necessary resources, causing a negative impact on operations that extends the scope beyond the poorly implemented software.

Weakness in software development practices may lower the cyber resiliency of the entire hosting system, resulting in degradation of organizationally defined levels of CIA. Examples include CWE-787 “Out-of-bounds Write,” CWE-125 “Out-of-bounds Read,” CWE-190 “Integer Overflow or Wraparound,” and CWE-476 “NULL Pointer Dereference,” which are all listed as among the 2023 CWE Top 25 weaknesses. These could result in violations of CIA and vulnerabilities in production applications. 

Complete Mediation

Complete mediation is embodied in the assurance that every request by a subject for accessing an object (especially for security-critical objects) must be vetted. That is, it should undergo a valid and effective authorization procedure. A system that relies on checking the permissions of a subject to an object only once can potentially invite attackers to exploit that system. Caching permissions can increase the performance of a system, but at the cost of allowing secured objects to be accessed.

For example, when a UNIX process tries to read a file, the operating system determines whether the process is allowed to read the file. If so, the process receives a file descriptor encoding the allowed access. Whenever the process wants to read the file, it presents the file descriptor to the kernel. The kernel then allows access. If the owner of the file disallows the process permission to read the file after the file descriptor is issued, the kernel still allows access. This scheme violates the principle of complete mediation, because the second access is not checked. The cached value is used, resulting in the denial of access being ineffective.

Open Design

The concept of open design dictates that the security of a system should not be dependent on secrecy of its design, its implementation, or its components. As an example, publishing an encryption algorithm for review by experts can in fact result in the identification of weaknesses in the algorithm which in turn results in increased security.

In this regard, noting that security through obscurity alone may not be achieved, Kerckhoff's principle suggests that so long as secrecy of the encryption key is maintained, the security of a cryptographic system should also be maintained, even if everything about the system is public knowledge. Basing security on open standards could potentially increase portability and interoperability as well.

MOSA

, previously known as Open Systems Architecture or Open Systems Approach, is a technical and business strategy for designing adaptable systems including software. MOSA requires that major interface points within systems are modular and embrace widely supported standards. To be MOSA-compliant, standards must support interoperability, scalability, and portability. MOSA standards are key to building a system capable of information-sharing across domains, with interchangeable hardware and software components.

Open-Source Software

libraries and other products are not inherently more or less secure just because the source code is freely available. Often the security of OSS software depends on the community rigor and bandwidth to provide updates throughout the life cycle of the source code. How many developers contribute to the project—one or one hundred? If only a limited number of maintainers exist is there any guarantee of timely patching or analysis for weaknesses? Further, who is allowed to commit changes to the source and what is the level of testing involved before commits are incorporated within the main source branch? Based on contributors, what is the level of confidence that source will be reasonably free of intentional or unintentional defects.

For example, Advanced Encryption Standard (AES) has been open source since 2001, and anyone can see with absolute transparency the inner workings of this encryption method. However, it is one of the most implemented and secure encryption methods still employed today.

Collaborative Design

In general, for any type of collaborative effort to be successful, its culture must exist, and collaborative software development is no different. This type of collaboration is often seen in open-source communities where developers contribute to projects with some level of autonomy.

Peer reviews are important in all software development projects in general, but in this type of environment these peer reviews would be critical.

From a security perspective, the ability to tap subjects from all over the world with different skills, experiences, knowledge bases, and thoughts can be invaluable. This has been evident from a number of well-known software products that are the result of crowdsourcing.

Least Common Mechanism

The concept of least common mechanisms involves minimizing the number of protection mechanisms that are common to multiple subjects. In so doing, the shared access paths that can be sources of unauthorized information exchange are reduced. For example, NIST Special Publication 800-44—Guidelines on Securing Public Web Servers notes that the ability for the web server process to access a back-end database, for instance, should not also enable other applications on the system to access the back-end database.

The Cybersecurity and Infrastructure Security Agency says to avoid having multiple subjects sharing mechanisms to grant access to a resource. For example, serving an application on the Internet allows both attackers and users to gain access to the application. Sensitive information can potentially be shared between the subjects via the mechanism. A different mechanism (or instantiation of a mechanism) for each subject or class of subjects can provide flexibility of access control among various users and prevent potential security violations that would otherwise occur if only one mechanism was implemented.

Compartmentalization

In the context of the least-common mechanism principle, restriction of user roles for the purpose of compartmentalization or isolation of the code/function is recommended. In other words, execution of different functions based on user roles is recommended in contrast to having a single function used by all roles.

Allow/Accept Lists

Allow/accept lists explicitly allows access to a particular resource. It is one method to address access control for subjects that may need to access an object. With allow/accept lists, a particular object is accessible while access to everything else is denied. In this way, attempts to access an object in a nonstandard way will be denied.

Psychological Acceptability

According to US-CERT, the security principle of psychological acceptability aims at maximizing the use and adoption of the security functionality in the software by ensuring that it is easy to use and transparent to the user. Ease of use and transparency are essential requirements for this security principle to be effective.

In this context, think about situations where security mechanisms inhibit accessibility to resources or reduce usability. Subjects likely will turn off such mechanisms or find ways to circumvent them.

Passwordless Authentication

Today, there are several methods that authenticate users. For decades, passwords (something you know) have been the dominant authentication method. Over time, new factors of authentication have been developed and leveraged in addition to passwords that create multifactor authentication. However, it is entirely possible to provide comparable security with more convenience by authenticating using one or more factors that do not include passwords. Users can authenticate using access to applications such as email, devices such as phones, one-time pass codes, or biometrics. Passwordless multifactor is possible by implementing combinations of something you have and something you are, for example.

CAPTCHA

Most users have come across the Completely Automated Public Turing Test to Tell Computers and Humans Apart (CAPTCHA) during authentication. CAPTCHA tests require the user to type letters or digits from a distorted image, or to click on all images in an array that fit a certain description (e.g., all images that contain traffic lights). CAPTCHA challenges can take a variety of different shapes and forms.

Several published vulnerabilities disclose how this control can be overcome. Reasons include:

  • An audio or visual image lacking sufficient distortion based on the original source image 
  • A question with a format that can be automatically recognized, such as a math question 
  • A question for which the number of possible answers is limited, such as birth years or favorite sports teams
  • A general-knowledge or trivia question for which the answer can be accessed using a data base, such as country capitals or popular actors
  • Other data associated with the CAPTCHA may provide hints about its contents, such as an image whose filename contains the word that is used in the CAPTCHA.12

Component Reuse

This concept is about promoting reusability of existing components, so that no newer vulnerabilities are introduced, and the attack surface is not increased. Software design should consider leveraging existing components.

One may also argue that centralizing functionality into business components that can be leveraged repeatedly is akin to putting all eggs in one basket. However, proper defense-in-depth and layered security measures can mitigate this.

Libraries and Common Controls

Tested and proven libraries and common programming controls should be reused and leveraged. It is better to reuse trusted systems and tools rather than implement an entirely new suite of products every time a new system must be designed.

One of the most common issues observed in software security code review is that development teams try to write their own cryptographic algorithms instead of using validated and proven standards such as the Advanced Encryption Standard (AES). Implementation of these customized cryptographic algorithms has, in most cases, been found to be the weakest link, thus disclosing sensitive information.
