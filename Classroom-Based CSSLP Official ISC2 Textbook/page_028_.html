Security Requirements Traceability Matrix (SRTM)

It is valuable to have the ability to not only capture all security requirements in one place but also trace from a security requirement to a test case or vice versa. NIST describes a Security Requirements Traceability Matrix (SRTM) as a “[m]atrix documenting the system’s agreed upon security requirements derived from all sources, the security features’ implementation details and schedule, and the resources required for assessment.28

As a traceability matrix, the SRTM can be used for any type of project and allow security requirements and security tests to be easily traced back to one another.

The creation of an SRTM involves mapping security requirements and corresponding verification methods (test cases). As shown in Table 3.2, a simple grid much like an Excel spreadsheet could be used for the documentation, with a column for each of the following:

  • Requirement identification number
  • Requirement description
  • Source of the requirement
  • Objective of the test
  • Verification method for the test

Each row is for a new security requirement, making an SRTM an easy way to view and compare the various requirements and tests that are needed in the software project. Sometimes organizations map RMF controls from NIST SP 800-53 using a SRTM.

28 National Institute of Standards and Technology; Computer Security Resource Center; https://csrc.nist.gov/glossary/term/security_requirements_traceability_matrix; retrieved June 2023.

There are three types of SRTM:

  • Forward traceability. Maps requirements to test cases.
  • Backward or reverse traceability. Maps test cases to requirements.
  • Bidirectional traceability. Maps requirements to test cases and vice versa.
